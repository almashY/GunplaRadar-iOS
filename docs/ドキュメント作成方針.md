# ドキュメント作成方針

---

## ドキュメント構成

```
docs/
├── ドキュメント作成方針.md       # 本ファイル
├── requirements/                # 要件定義
│   ├── README.md                # 概要・機能一覧・非機能要件
│   ├── F0x_機能名.md            # 機能ごとの要件
│   ├── screens.md               # 画面一覧
│   └── data_model.md            # ※詳細設計へ移動予定
└── basic_design/                # 基本設計
    └── F0x_機能名/
        ├── 01_対応方針.md
        ├── 02_分析クラス図.md
        └── 03_分析シーケンス図.md
```

---

## 要件定義（requirements/）

### 目的
「何を作るか」を定義する。実現方法（HOW）は記載しない。

### 作成ルール

| ルール | 内容 |
|--------|------|
| 粒度 | ユーザーが何を操作でき、何が得られるかの記述にとどめる |
| データ | エンティティ名と概念的な属性のみ記載。型・フィールド名は詳細設計で定義する |
| 分割単位 | 機能（F01〜）ごとに1ファイル。横断的な内容は `README.md` に集約する |
| 表記 | ユーザー視点の言葉で記述する（例：「登録する」「表示する」） |

---

## 基本設計（basic_design/）

### 目的
「どのように作るか」をドメインレベルで設計する。実装技術（SwiftUI・SwiftDataなど）の詳細はここでは扱わない。

### フォルダ構成ルール

- 機能（F01〜）ごとにフォルダを作成する
- フォルダ名は `requirements/` の機能名と統一する
- 各フォルダに必ず `01_対応方針.md`・`02_分析クラス図.md`・`03_分析シーケンス図.md` の3ファイルを作成する

---

### 01_対応方針.md

#### 目的
その機能をドメインレベルでどう解決するかの方針を記述する。

#### 記載項目

| 項目 | 内容 |
|------|------|
| ドメイン概要 | 機能の目的とドメインの中心的な概念を1〜2文で説明 |
| ドメインの責務 | クラス・役割の対応表 |
| 方針 | 各責務ごとの設計判断・ルール・制約 |
| 他機能との連携 | 参照・依存する機能とその内容 |

#### 記載ルール
- 実装ライブラリ・フレームワーク名は原則記載しない（インフラ層に委譲する旨のみ触れる）
- 「〜する」ではなく「〜とする」「〜で担う」などの設計決定の文体で記述する

---

### 02_分析クラス図.md

#### 目的
ドメインを構成するクラスとその関係を可視化する。

#### 記載ルール

| ルール | 内容 |
|--------|------|
| 形式 | Mermaid の `classDiagram` を使用する |
| クラス名 | すべて日本語で記載する（例：ガンプラアイテム、店舗分析サービス）。実際の実装クラス名は詳細設計で定義する |
| ステレオタイプ | `<<画面>>`・`<<インターフェース>>`・`<<列挙型>>`・`<<値オブジェクト>>` を適宜付与する |
| 属性・メソッド | 日本語で記載する。型表記は省略可。分析レベルで必要なもののみ記載し getter/setter は不要 |
| 関連線 | 画面→サービス・リポジトリ、サービス→リポジトリ・値オブジェクト、リポジトリ→エンティティ の繋がりをすべて記載する |
| ラベル | 関連の意図が自明でない場合は線にラベルを付与する |
| 多重度 | 集約関係（1対多）の場合は多重度を記載する（例：`"1" --> "0..*"`）|
| 外部サービス | DBやAPIサーバーなど外部サービスを利用する場合はクラスとして登場させ、`<<外部システム>>` ステレオタイプを付与する。どのクラス（リポジトリ等）からアクセスするかを関連線で明示する |

---

### 03_分析シーケンス図.md

#### 目的
主要なユーザー操作フローにおけるオブジェクト間のメッセージのやり取りを可視化する。

#### 記載ルール

| ルール | 内容 |
|--------|------|
| 形式 | Mermaid の `sequenceDiagram` を使用する |
| 参加者 | `actor ユーザー`・画面・サービス・リポジトリ（すべて日本語）・インフラ（括弧書きで補足）を記載する |
| 粒度 | ドメインレベルのやり取りを記述する。UI内部の詳細（ボタンのスタイル変更など）は不要 |
| フロー選定 | 機能ごとに主要な操作フロー（登録・表示・更新・削除）をそれぞれ1つのシーケンスとして記載する |
| 分岐 | `alt` / `else` を使って条件分岐を表現する |
| インフラ委譲 | 通知スケジューラ・MapKitなどインフラ層の処理は参加者名に `（インフラ）` と付記し、ドメインとの境界を明示する |
| 外部サービス | DBやAPIサーバーなど外部サービスを利用する場合は参加者として登場させ、参加者名に `（外部）` と付記する。どの参加者からアクセスするかをメッセージ矢印で明示する |
