# 実装方針

---

## テスト駆動開発（TDD）方針

### 基本ルール

実装依頼を受けた場合は、テスト駆動開発（TDD）で進める。

1. **テストが必要か判断する**
   追加・変更するメソッドやクラスに対して、ユニットテストが必要かを判断する（判断基準は後述）
2. **テストケースを洗い出し、ドキュメントに記録する**
   `docs/tests/` 配下の対象機能ファイルにテストケースを記載する
3. **ユニットテストコードを先に実装する**
4. **テストが失敗することを確認する**（Red）
5. **実装コードを書く**
6. **テストが通ることを確認する**（Green）

---

### ユニットテストの対象判断

| 対象 | テスト作成 | 理由 |
|------|-----------|------|
| ViewModel | ○ | ビジネスロジック・状態管理を含む |
| Service | ○ | ドメインロジックの中核 |
| Repository 実装クラス | ○ | データ永続化の正確性を保証 |
| Utility / Calculator | ○ | 純粋な計算ロジックを含む |
| View（SwiftUI） | ✕ | 画面で確認できる内容は人間が手動で確認するため不要 |

---

### テストケースの種別

| 種別 | 内容 |
|------|------|
| 正常系 | 期待通りに処理が完了するケース |
| 異常系 | エラー・例外が発生するケース |
| 境界値 | 空配列・nil・最大値・最小値など境界となる入力値のケース |

---

### テストドキュメントの管理

- ユニットテストケースと手動確認項目は、どちらも **`docs/tests/`** 配下に機能ごとのファイルとして記録する
- ユニットテストは TDD の手順に沿ってコードに落とし込む
- 手動確認項目（画面で確認できる内容）は `docs/tests/` のドキュメントに記載し、人間がテスト手順に従って確認する

---

## ドキュメントフォーマット

### ユニットテストケース（`docs/tests/`）

```markdown
## クラス名

### メソッド名

| # | テストケース | 入力 | 期待結果 | 種別 |
|---|------------|------|---------|------|
| 1 | ○○の場合に△△になること | ... | ... | 正常系 |
| 2 | ○○が空の場合にエラーになること | ... | ... | 異常系 |
| 3 | 件数が 0 件の場合に空配列を返すこと | [] | [] | 境界値 |
```

### 手動確認項目（`docs/tests/`）

```markdown
## 手動確認項目

### 確認手順: ○○機能

| # | 操作手順 | 期待結果 | 確認 |
|---|---------|---------|------|
| 1 | ～をタップする | ～が表示される | □ |
| 2 | ～を入力して保存する | ～が登録される | □ |
```

## ドキュメント更新方針

実装を行った場合は、変更内容に応じて関連ドキュメントを合わせて修正する。

| 実装内容 | 更新対象ドキュメント |
|---------|------------------|
| クラス・メソッドの追加・変更・削除 | `detailed_design/` の詳細クラス図・詳細シーケンス図 |
| ドメインロジックの変更 | `basic_design/` の分析クラス図・分析シーケンス図・対応方針 |
| データモデルの変更 | `detailed_design/データモデル設計書.md` |
| 画面レイアウト・UI仕様の変更 | `detailed_design/画面設計書.md` |
| エラーハンドリングの変更 | `detailed_design/エラーハンドリング設計書.md` |
| 通知ロジックの変更 | `detailed_design/通知設計書.md` |
| テストケースの追加・変更 | `tests/` の対象機能ファイル |
